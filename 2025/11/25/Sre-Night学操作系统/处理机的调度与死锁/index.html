处理机的调度与死锁

发表于2025-11-25|更新于2025-11-25|Sre-Night学操作系统
|总字数:4.2k|阅读时长:13分钟|浏览量:

第3章 处理机调度与死锁

3.1 处理机调度概述

  • 处理机调度层次

    • 高级调度（长程调度/作业调度）
    • 低级调度（短程调度/进程调度）
    • 中级调度（中程调度/内存调度）

  • 高级调度

    • 调度对象：作业
    • 定义：根据算法将外存后备队列中的作业调入内存，创建进程并分配资源，将新进程排入就绪队列等待调度。
    • 应用：多道批处理系统

  • 中级调度

    • 功能：将暂不运行的进程调至外存等待；将外存上急需运行的进程调入内存运行（即“对换”功能）
    • 说明：将在第5章 存储器管理中介绍

  • 低级调度

    • 调度对象：进程
    • 定义：根据调度算法，决定就绪队列中的哪个进程获得处理机
    • 应用：多道批处理、分时和实时OS

  • 进程调度的任务和方式

    • 进程调度的任务：保存处理机现场信息；按算法选取进程；把处理器分配给进程
    • 进程调度机制（调度程序三部分）：
      • 排队器：将就绪进程插入相应就绪队列
      • 分派器：将选定的进程移出就绪队列
      • 上下文切换器：进行新旧进程上下文切换

  • 进程调度的方式

    • 非抢占方式：一旦分配处理机，进程一直执行到完成或阻塞，不允许抢占。
    • 抢占方式：调度程序可暂停正在执行的进程，重新分配处理机（现代OS广泛采用）。
      • 优先级原则：高优先级新进程抢占当前进程处理机
      • 短作业优先原则：短作业抢占当前较长作业的处理机
      • 时间片原则：进程按时间片运行，时间片用完后重新调度

  • 处理机调度算法的目标

    • 共同目标：资源利用率、公平性、平衡性、策略强制执行

    • 批处理系统目标：平均周转时间短、系统吞吐量高、处理机利用率高

    • 分时系统目标：响应时间快、均衡性

    • 实时系统目标：截止时间的保证、可预测性

    • CPU利用率公式：
      $$
      \text{CPU利用率} = \frac{\text{CPU有效工作时间}}{\text{CPU有效工作时间}+\text{CPU空闲等待时间}}
      $$

  • 评价指标

    • 周转时间：完成时间 - 到达时间
      $$
      T = \frac{1}{n}\left( \sum_{i=1}^{n} T_i \right)
      $$

      • 平均周转时间：作业从提交到完成为止的时间间隔的平均值

      • 带权周转时间：作业周转时间( T )与系统服务时间( TS )之比
        $$
        W = \frac{1}{n}\left( \sum_{i=1}^{n} \frac{T_i}{T S_i} \right)
        $$

    • 响应时间：从用户提交请求到系统首次显示处理结果的时间

    • 等待时间：周转时间 - 运行时间（进程调度中，进程在就绪队列等待调度的总时间）

3.2 调度算法

  • 作业调度算法

    • 先来先服务调度算法(FCFS)
    • 短作业优先调度算法(SJF)
    • 优先级调度算法(PR)
    • 高响应比优先调度算法(HRN)
    • 说明：FCFS、SJF、PR既可用于作业调度，也可用于进程调度

  • 进程调度算法

    • 先来先服务调度算法(FCFS)
    • 短作业优先调度算法(SJF)
    • 优先级调度算法(PR)
    • 时间片轮转调度算法(RR)
    • 多级队列调度算法
    • 多级反馈队列调度算法
    • 基于公平原则的调度算法

  • 先来先服务（FCFS）调度算法

    • 规则：按作业到达先后次序调度

    • 举例：作业

    • 进程调度举例

  • 短作业优先（SJF）调度算法

    • 规则：从后备队列中选择估计运行时间最短的作业（进程调度中选择下一次CPU区间最短的进程）

    • 模式：非抢占式SJF、抢占式SJF（最短剩余时间优先调度）

    • 特点：对一组指定进程，能给出最短的平均等待时间

    • 缺点：只能估算进程运行时间（不准确）；对长作业不利；人机无法交互；未考虑作业紧迫程度

  • 优先级调度算法（PR）

    • 规则：基于作业/进程的紧迫程度，由外部赋予优先级，按优先级调度

    • 类型：非抢占式、抢占式；静态优先级（创建时确定，运行期不变）、动态优先级（随运行/等待时间改变）

    • 非抢占式优先级调度举例：

    • 优点：实现简单；考虑了进程紧迫程度；灵活，可扩充算法

    • 问题：低优先级进程可能“饥饿”（永远得不到运行）

    • 解决方法：老化（随等待时间延长提高优先级）

    • 高响应比优先调度算法（PR特例）：
      $$
      优先级 = ( \frac{\text{等待时间}+\text{要求服务时间}}{\text{要求服务时间}} = \frac{\text{响应时间}}{\text{要求服务时间}} )，既考虑等待时间，又考虑运行时间
      $$

  • 时间片轮转（RR）调度算法

    • 设计：为分时系统设计，类似FCFS但增加抢占；时间片为小单位CPU时间（10-100毫秒）

    • 规则：为每个进程分配不超过一个时间片的CPU时间，时间片用完后插入就绪队列末尾，循环执行

    • 时间片大小的确定：需考虑系统响应时间要求、就绪进程数目、系统处理能力；一般原则：时间片 > 进程上下文切换时间

  • 多级队列调度算法

    • 规则：就绪队列分为多个（如前台交互队列、后台批处理队列），每个队列有自己的调度算法（前台RR、后台FCFS）；调度在队列间进行（固定优先级调度、时间片轮转调度）

  • 多级反馈队列调度算法

    • 规则：进程能在不同队列间移动；克服其他调度算法的局限性（如短进程优先忽略长进程）
    • 优点：不必事先知道进程所需执行时间；可满足各种进程的需要

  • 基于公平原则的调度算法

    • 目标：主要考虑调度的公平性，保证调度算法的性能保证（如确保进程的公平性，处理时间为1/n）

3.3 实时调度

  • 实时调度概述
    • 调度对象：实时任务，关联截止时间（硬实时HRT任务、软实时SRT任务）
  • 实现实时调度的基本条件
    1. 提供必要的信息：就绪时间、开始/完成截止时间、处理时间、资源要求、优先级
    2. 系统处理能力强：单/多处理机系统需满足任务处理需求
    3. 采用抢占式调度机制
    4. 采用快速切换机制：对中断快速响应、快速任务分派
  • 实时调度算法分类
    • 按实时任务性质：HRT调度算法、SRT调度算法
    • 按调度方式：非抢占式调度算法、抢占式调度算法
  • 非抢占式调度算法
    • 非抢占式轮转调度算法：响应时间数秒至数十秒，用于要求不太严格的实时控制系统
    • 非抢占式优先调度算法：响应时间数秒至数百毫秒，用于有一定要求的实时控制系统
  • 抢占式调度算法
    • 基于时钟中断的抢占式优先级调度：响应时间几十毫秒至几毫秒，用于大多数实时系统
    • 立即抢占的优先级调度：响应时间几毫秒至几百微秒，用于有严格时间要求的实时系统
  • 最早截止时间优先（EDF）调度算法
    • 规则：根据任务截止时间确定优先级，截止时间越早优先级越高；既可用于抢占式，也可用于非抢占式（非抢占式用于非周期实时任务，抢占式用于周期实时任务）
  • 最低松弛度优先（LLF）算法
    • 规则：根据任务的紧急程度（松弛度）确定优先级，紧急程度越高（松弛度越低）优先级越高；松弛度 = 必须完成时间 - 本身运行时间 - 当前时间；主要用于抢占式调度
    • 例子：周期任务A（每20ms执行一次，执行时间10ms）、任务B（每50ms执行一次，执行时间25ms）
  • 优先级倒置现象
    • 现象：采用优先级调度和抢占方式时，高优先级进程被低优先级进程延迟或阻塞
    • 解决方法：制定规则（如低优先级进程执行后不允许抢占）、建立动态优先级继承

3.4 Linux进程调度

  • 调度架构
    • 默认调度算法：完全公平调度（CFS）算法
    • 调度器类：允许不同调度算法并存，每个类有特定优先级；总调度器按调度器类优先级依次调度；调度器类内部使用所选算法调度
    • 调度器类默认优先级顺序：Stop_Task > Real_Time > Fair > Idle_Task
  • 调度器类详情
    1. Stop_Task：优先级最高的特殊任务，用于暂停目标CPU上的所有其他任务，执行内核关键操作（如CPU热插拔、内核同步）
    2. Real_Time：实时任务采用SCHED_FIFO（先进先出）或SCHED_RR（时间片轮转）调度策略，优先级1-99，用于严格实时响应场景（如工业机器人控制、专业音频处理）
    3. Fair：由CFS调度，是普通用户进程的默认调度方式；根据进程“nice值”（-20~19，值越小优先级越高）分配CPU时间，实现“相对公平”（如Chrome浏览器进程、终端命令grep/find）
    4. Idle_Task：每个CPU核心有一个Idle_Task（进程名swapper，PID=0），CPU无其他可运行任务时执行，主要执行低功耗操作（如系统空闲时的CPU空闲占比%id）
  • 进程调度策略
    • 普通进程调度：采用SCHED_NORMAL策略，优先级与友好值（-20~+19）有关，数值越低优先级越高
    • 实时进程调度：优先级高于普通进程；SCHED_FIFO进程就绪后一直执行到阻塞或主动放弃；SCHED_RR进程时间片耗尽后需重新调度

3.5 死锁概述

  • 死锁概念
    • 死锁（Deadlock）：多个进程因争夺资源造成的僵局，若无外力作用，进程将永远无法推进
  • 资源问题
    • 可重用性资源和可消耗性资源：
      • 可重用性资源：一次只能分配给一个进程，遵循“请求-使用-释放”流程（如大部分资源）
      • 可消耗性资源：由进程动态创建和消耗（如进程间通信的消息）
    • 可抢占性和不可抢占性资源：
      • 可抢占性资源：进程获得后可被其他进程/系统抢占（如CPU、主存区）
      • 不可抢占资源：分配后只能由进程自行释放（如打印机、磁带机）
  • 死锁原因
    1. 竞争不可抢占性资源引起死锁
    2. 竞争可消耗性资源引起死锁
    3. 进程推进顺序不当引起死锁（合法推进顺序 vs 非法推进顺序）
  • 死锁定义
    • 死锁：一组等待的进程，每个进程都持有资源且等待组中其他进程持有的资源
  • 产生死锁的必要条件
    1. 互斥：一段时间内某资源只能被一个进程占用
    2. 请求和保持：进程至少持有一个资源，同时等待其他进程持有的资源
    3. 不可抢占：资源只有在持有它的进程完成任务后才能自由释放
    4. 循环等待：等待资源的进程之间存在循环（如P0等P1的资源，P1等P2的资源……Pn等P0的资源）
  • 处理死锁的方法
    1. 确保系统永远不进入死锁状态：死锁预防、死锁避免
    2. 允许系统进入死锁状态，然后恢复：死锁检测、死锁恢复
    3. 忽略问题：假装系统从未出现死锁（如UNIX采用此方法）

3.6 预防死锁

  • 策略：破坏死锁的四个必要条件中的一个或几个
  • 破坏“请求和保持”条件
    • 要求进程执行前一次性申请全部资源，只有无占有资源时才可分配资源
    • 改进：进程获得运行初期所需资源后开始运行，运行中逐步释放已分配资源，再请求新资源
  • 破坏“不可抢占”条件
    • 若进程申请资源未实现，需释放所有占有的资源；进程重新得到旧资源时可重新开始
  • 破坏“循环等待”条件
    • 对所有资源类型进行线性排序并赋予不同序号，要求进程按递增顺序申请资源

3.7 避免死锁

  • 基本思想
    • 要求每个进程声明所需资源的最大数；死锁避免算法动态检查资源分配状态，确保不会出现循环等待
  • 安全状态
    • 定义：存在一个安全序列（进程序列<P1,P2,…,Pn>，每个进程Pi的资源请求可被满足），则系统处于安全状态
    • 基本事实：
      • 系统在安全状态→无死锁
      • 系统不在安全状态→可能死锁
      • 死锁避免→确保系统永远不进入不安全状态
  • 银行家算法
    • 适用场景：资源有多个实例；进程需事先声明最大资源用量；进程请求资源时可能等待；进程获得所有资源后需在有限时间内释放
    • 数据结构
      • Available：长度为m的向量，表示每种资源的可用实例数
      • Max：n×m矩阵，表示每个进程对每种资源的最大请求数
      • Allocation：n×m矩阵，表示每个进程当前分配的资源实例数
      • Need：n×m矩阵，表示每个进程还需的资源实例数（Need[i,j] = Max[i,j] - Allocation[i,j]）
    • 安全性算法
      1. 初始化Work = Available，Finish[i] = false（i=1..n）
      2. 查找满足Finish[i]=false且Need[i] ≤ Work的进程i；若不存在，跳至步骤4
      3. Work = Work + Allocation[i]，Finish[i] = true；跳至步骤2
      4. 若所有Finish[i]=true，则系统处于安全状态
    • 资源请求处理流程
      1. 检查Request[i] ≤ Need[i]，否则报错（请求超出声明最大值）
      2. 检查Request[i] ≤ Available，否则进程等待
      3. 假设分配资源：Available = Available - Request[i]；Allocation[i] = Allocation[i] + Request[i]；Need[i] = Need[i] - Request[i]
      4. 执行安全性算法，若系统安全则分配，否则恢复原资源分配状态
    • 例子：5个进程（P0-P4），3种资源（A:10、B:5、C:7）
      • 时刻T的状态：
        • Allocation：P0(0,1,0)、P1(2,0,0)、P2(3,0,2)、P3(2,1,1)、P4(0,0,2)
        • Max：P0(7,5,3)、P1(3,2,2)、P2(9,0,2)、P3(2,2,2)、P4(4,3,3)
        • Available：(3,3,2)
      • 安全序列：<P1,P3,P4,P2,P0>，系统处于安全状态

3.8 死锁的检测与解除

  • 死锁检测和解除
    • 若系统未采取限制性措施，需提供检测和解除死锁的手段，包括：保存资源请求和分配信息；提供算法检测系统是否进入死锁状态
  • 资源分配图
    • 组成：顶点集V（进程集P、资源集R）和边集E（请求边P→R、分配边R→P）
    • 死锁判断：若资源分配图中存在循环，且每种资源类型只有一个实例，则死锁发生；若资源类型有多个实例，可能死锁
  • 资源分配图的简化
    • 步骤：找出既不阻塞又非独立的进程p，使其获得所需资源并运行完毕，释放所有资源（消去p的请求边和分配边）；重复此过程，若能消去所有边则图可完全简化（无死锁），否则不可完全简化（存在死锁）
  • 死锁定理
    • 系统S为死锁状态的充分必要条件：S的资源分配图是不可完全简化的
  • 死锁检测算法
    • 类似于银行家算法，基于资源分配图简化
    • 步骤：初始化Work = Available，Finish[i] = false；查找满足Finish[i]=false且Request[i] ≤ Work的进程i，更新Work和Finish；若存在Finish[i]=false，则进程Pi是死锁的
  • 死锁的解除
    • 抢占资源：从一个或多个进程中抢占足够资源给饥饿进程，解除死锁状态
    • 终止进程：终止系统中一个或多个死锁进程，直到打破循环（如终止所有死锁进程、终止一个进程）
    • 进程回退：中断所有死锁进程，或逐个中断直到死锁消失；选择中断顺序时需考虑代价（如进程优先级、运行时间、资源需求等）

第4章 进程同步一、进程同步概念 主要任务：使并发执行的诸进程之间能有效地共享资源和相互合作，从而使程序的执行具有可再现性。 进程间的制约关系： 间接相互制约关系（互斥关系）：进程互斥使用临界资源 直接相互制约关系（同步关系）：进程间相互合作 临界资源： 系统中某些资源一次只允许一个进程使用，称这样的资源为临界资源或互斥资源或共享变量。 诸进程间应采取互斥方式，实现对这种资源的共享。 临界区 临界区：进程中涉及临界资源的代码段 进入区：用于检查是否可以进入临界区的代码段 退出区：将临界区正被访问的标志恢复为未被访问标志 剩余区：其他代码 访问临界资源的循环进程描述： While(TRUE){ 进入区 临界区 退出区 剩余区} 同步机制应遵循的准则 空闲让进：当无进程处于临界区，应允许一个请求进入临界区的进程立即进入自己的临界区 忙则等待：已有进程处于其临界区，其它试图进入临界区的进程必须等待 有限等待：等待进入临界区的进程不能死等 让权等待：不能进入临界区的进程，应释放CPU（如转接到阻塞状态） 进程同步机制 软件...
